02 · Tipografia & Arquitetura de Informação

Hierarquia tipográfica para ambientes de alta densidade

Três famílias com funções distintas. Space Grotesk ancora a identidade nos títulos com personalidade geométrica. IBM Plex Sans oferece legibilidade neutra e prolongada em formulários e tabelas. IBM Plex Mono garante alinhamento perfeito em timestamps, números de série e hashes criptográficos.

Headings / Display
Space Grotesk

Títulos de páginas, seções do painel, labels de cards de KPI. Geométrica e moderna — comunica autoridade técnica sem frieza excessiva. Usada em peso 300–700.

Body / Interface
IBM Plex Sans

Corpo de texto, labels de formulário, conteúdo de tabelas, mensagens de feedback. Humanista e altamente legível em tamanhos pequenos. Leitura confortável em telas por períodos prolongados.

Dados / Monospaced
IBM Plex Mono

Números de série, timestamps, hashes, IDs de movimentação, logs de auditoria. Alinhamento fixo evita drift visual em tabelas de dados — o olho lê colunas numéricas com zero esforço.

03 · Diretrizes para Iconografia Própria

Construção de ícones customizados

Ícones são construídos sob um grid rigoroso para garantir consistência visual em toda a plataforma. Estilo: vazado (outline), cantos arredondados, traço uniforme. Sem ícones sólidos preenchidos — o estilo outline reduz peso visual em interfaces densas e mantém legibilidade em tamanhos pequenos.

Grid Base
24×24px
Canvas de construção. Área de segurança interna: 20×20px. Nunca extrapolar a área de segurança.
Stroke Weight
1.5px
Traço único em toda a biblioteca. Não misturar 1px e 2px no mesmo conjunto.
Cantos (Corner)
Round
stroke-linecap: round · stroke-linejoin: round. Cria percepção de amigabilidade técnica, não fria.
Estilo
Outline
Sempre vazado. Preenchimento apenas em estados de erro crítico ou status de alerta para máximo destaque semântico.
Pixel Snap
Obrigatório
Linhas horizontais e verticais devem alinhar ao pixel. Evita blur em resoluções 1x.
Cor Padrão
currentColor
Ícones herdam a cor do texto pai. Nunca hardcode de cor no SVG — facilita estados hover e temas.
Entregar Cartão
Cartão com seta de saída para a direita. Modelo mental: cartão passando de mãos.
Registrar Devolução
Cartão com seta retornando para a esquerda. Espelho do ícone de entrega — cria associação imediata de inversão da operação.
Consultar Status
Lupa com relógio interno. Evoca busca no tempo — histórico + tempo real. Mais específico que lupa simples.
Gerar Comprovante
Documento com check. O operador já usa "comprovante" como conceito mental — o check confirma que é o documento de confirmação, não apenas um arquivo.
Criar Notificação
Sino com faísca de "novo". O sino é reconhecível como notificação; a faísca de criação diferencia de "ver notificações".
04 · Padrões de UX Writing

Linguagem da interface e sistema de feedback

A linguagem da interface deve refletir o domínio operacional do administrador. Termos técnicos do código não chegam à UI. Todo texto é claro, direto e nunca culpa o usuário.

Glossário Fechado da Interface

Termo Técnico (Código) Evitar na UI Usar na UI Justificativa
Beneficiario Usuário / Cliente Beneficiário "Beneficiário" é o termo oficial do domínio de programas sociais. "Usuário" cria ambiguidade com o administrador do sistema. "Cliente" traz conotação comercial incorreta.
Administrador Admin / User / Operador Administrador Mantém o vocabulário dos RF/RNF. Consistência entre documentação técnica e interface reduz curva de aprendizado.
Movimentacao Transação / Evento / Log Movimentação "Transação" tem conotação financeira. "Movimentação" reflete o processo físico do cartão — mais preciso ao modelo mental do operador.
CartaoAuxilio Card / Token / Item Cartão Terminologia familiar ao operador que lida com o objeto físico diariamente. Evita jargão técnico.
RegistroAuditoria Log / Trace / Histórico Registro de Auditoria Mantém o peso institucional necessário. "Log" é jargão de TI invisível ao operador.
condicaoCartao State / Flag / Bool Condição do Cartão Linguagem natural que qualquer operador entende sem treinamento adicional.
authDoisFatores 2FA / MFA / OTP Verificação em dois passos Familiar ao usuário de plataformas de e-mail/banco. Evita siglas que geram dúvida.

Padrões de Escrita para Botões de Ação (CTAs)

Operação (Diagrama de Classes) Label do Botão Preview Regra de Escrita
registrarEntrega() Confirmar Entrega Confirmar Entrega Verbo de confirmação + substantivo do domínio. Evitar "Registrar" como verbo de botão — é jargão de sistema.
registrarDevolucao() Confirmar Devolução Confirmar Devolução Mesmo padrão da entrega. Consistência reduz carga cognitiva em operações inversas.
gerarRegistro() Gerar Comprovante PDF Gerar Comprovante PDF Verbo de ação + objeto + formato. Mencionar "PDF" define expectativa do resultado.
transferirResponsabilidade() Transferir Cartão Transferir Cartão Ação secundária — estilo ghost. "Responsabilidade" é jargão. O operador pensa em "transferir o cartão".
criarNotificacao() Criar Notificação Criar Notificação "Criar" é aceito quando o objeto já é familiar (notificação). Diferente de "Configurar" — que sugere algo mais técnico.
validarPreRequisitos() — falha Cancelar Cancelar Operação Ação destrutiva — tom de alerta suave. Nunca "Deletar" ou "Apagar" para operações de cancelamento.
